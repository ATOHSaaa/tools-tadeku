# IWE — Integrated Writing Environment 設計書

小説執筆のための総合執筆環境。IDE(統合開発環境)になぞらえた IWE(統合執筆環境)。
メモ → プロット → 原稿という創作の流れを 1 つの画面で支援し、Git によるバージョン管理と
GitHub 連携、複数形式のプレビューを備える。

この文書は Cursor の Composer に実装を依頼するための仕様書である。
「8. 実装ガイド」のフェーズ順に実装すること。

---

## 1. 前提・アーキテクチャ

- **形態**: tools-tadeku リポジトリ内の他ツールと同じ、完全クライアントサイドの静的 Web アプリ。
  サーバー・アカウント登録なし。GitHub Pages(tools.tadeku.net)でそのまま配信する。
- **配置**: `iwe/` ディレクトリ。エントリは `iwe/index.html`。
  規模が大きいため単一 HTML に押し込まず、`iwe/js/*.js` に分割して `<script>` で読み込む
  (バンドラーは使わない)。IWE 固有でないロジックは `shared/iwe-*.js` に置いてもよい。
- **対象ユーザー**: 不特定多数(tadeku.net の公開ツール)。初回訪問で説明なしに使い始められること。
  Git の知識を前提にしない UI 語彙を使う(後述)。
- **外部ライブラリ**(すべて CDN、ビルド不要):
  - [isomorphic-git](https://isomorphic-git.org) v1 — ブラウザ内 Git 実装
  - [@isomorphic-git/lightning-fs](https://github.com/isomorphic-git/lightning-fs) v4 — IndexedDB 上の仮想ファイルシステム
  - 読み込みは `https://esm.sh/isomorphic-git@1` / `https://esm.sh/@isomorphic-git/lightning-fs@4`
    のような ESM import(`type="module"` スクリプト内)。動かない場合は unpkg の UMD ビルドに切り替えてよい
- **既存共有資産の再利用**(詳細は各節):
  - `shared/nyuko-engine.js` + `shared/nyuko-data.js` — 縦書き組版・記法パーサ(`window.NyukoEngine`, `window.NyukoData`)
  - `genko/index.html` 内の原稿用紙グリッド・禁則処理・縦組み字形変換ロジック
  - `shared/cast-engine.js` — 固有名詞抽出(`window.TadekuCastEngine`)
  - `shared/site-link.js` / `site-link.css` — サイト共通のパンくずブランド表示
- **サイト共通の作法**: 他ツールの `index.html` に倣い、Google Fonts・gtag・`shared/site-link.js`
  を組み込む。トップページ `index.html` と `sitemap.xml` への追加は実装完了後に別途行う(この設計のスコープ外)。

## 2. データモデルと保存

### 2.1 基本概念

**作品(work)** が最上位の単位。作品ごとに独立した Git リポジトリを 1 つ持つ。
作品の中に 4 種類のドキュメントがある。すべて **Markdown + YAML frontmatter のテキストファイル**
として仮想ファイルシステム(LightningFS)に保存する。
「すべてがファイル」であることで Git 管理・GitHub 同期・全文検索と自然に整合する。

| 種類 | ディレクトリ | 役割 |
|---|---|---|
| メモ | `memos/` | ネタ・断片・思いつき。1 ファイル 1 ネタ |
| プロット | `plots/` | 構成案。メモを起点にしても、ゼロから書いてもよい |
| 原稿 | `manuscript/` | 小説本文。章ごとに 1 ファイル |
| 人物・設定 | `characters/` | 登場人物シート・世界観資料 |

### 2.2 仮想ファイルシステムのレイアウト

LightningFS のインスタンス名(= IndexedDB 名)は **`iwe-fs`** とする
(既存の `tadeku-tools` / `tadeku-etude` / `stock` と衝突させないこと)。

```
/works/<workId>/            ← ここが Git リポジトリのルート
  iwe.json                  ← 作品メタ(下記)
  memos/雨の匂いの話.md
  plots/第一部構成.md
  manuscript/001-序章.md
  manuscript/002-出会い.md
  characters/主人公.md
```

- `<workId>` は作成時に生成する短いランダム ID。表示名は `iwe.json` に持つ
  (ファイルパスに作品名を使うとリネームが Git 履歴上の全ファイル移動になるため)。
- `manuscript/` 内は `NNN-タイトル.md` の連番プレフィックスで章順を表現する。
  章の並べ替えはリネームで行う。
- `iwe.json` の内容: `{ "title": 作品名, "createdAt": ..., "targetChars": 目標文字数 | null }`
- 作品一覧(workId のリスト)は `/works/` 直下のディレクトリ走査で得る。別台帳は持たない。

### 2.3 ドキュメントのファイル形式

```markdown
---
title: 出会い
type: manuscript        # memo | plot | manuscript | character
source: plots/第一部構成.md   # 昇格元(任意)
---
本文……
```

- frontmatter は最小限に保つ。`title` と `type` は必須、他は任意。
- 人物シート(`type: character`)のみ追加フィールドを持つ:
  `name`(正式名)、`aliases`(呼称・あだ名の配列)、`notes` は本文に自由記述。

### 2.4 「昇格」フロー

- メモの画面に「このメモからプロットを作る」ボタン、プロットの画面に
  「このプロットから原稿を書き始める」ボタンを置く。
- 押すと新しいファイルを対応ディレクトリに作成し、frontmatter に `source:` として元ファイルの
  相対パスを記録、本文には元の内容を引用として挿入した状態でエディタを開く。
- 逆方向の参照(このメモから生まれたプロット一覧など)は、全ファイルの `source:` を走査して
  表示する(後述 7.3 のバックリンクと同じ仕組み)。
- 起点なしの新規作成も、各ディレクトリの「+ 新規」ボタンから当然できる。

### 2.5 自動保存

- エディタの入力は debounce(1 秒程度)で LightningFS に書き戻す。明示的な保存操作は不要。
- Git のコミットは自動保存と独立(ユーザー操作またはスナップショット提案による。3.2 参照)。

## 3. Git 機能

### 3.1 ローカル履歴(isomorphic-git)

- 作品作成時に `git.init()` で `/works/<workId>` をリポジトリ化。初期コミットまで自動で行う。
- UI 上は「Git」という語を前面に出さず、**「スナップショット」** と呼ぶ。
  - 「スナップショットを保存」= `git.add`(全ファイル)+ `git.commit`。メッセージは任意入力
    (空なら「2026-07-05 12:34 のスナップショット」のような自動文言)。
  - 履歴パネル: `git.log` をリスト表示。各項目から「この時点の内容を見る」「この時点に戻す」。
  - 差分表示: 選択した 2 時点(または現在との比較)について、ファイルごとのテキスト差分を表示。
    差分アルゴリズムは行単位の LCS を自前実装するか、小さな diff ライブラリ(jsdiff)を CDN で使う。
  - 「この時点に戻す」= `git.checkout` でワークツリーを復元し、その状態を新しいコミットとして
    積む(履歴を巻き戻すのではなく前進で戻す。reset --hard は使わない)。
- ブランチは v1 では扱わない。`main` 1 本。

### 3.2 スナップショットの促し

- 未コミットの変更が一定量(例: 2000 字以上の差分)たまったら、ステータスバーに
  「スナップショットしませんか」の控えめなバッジを出す。強制はしない。

### 3.3 GitHub 連携

Git Smart HTTP(clone/push)は GitHub が CORS を許可しておらず外部プロキシ
(cors.isomorphic-git.org 等)への依存が必要になるため **使わない**。
代わりに **GitHub REST API(Git Data API)** で同期を実装する。REST API は CORS 対応済み。

- **認証**: ユーザーが設定画面で Fine-grained Personal Access Token を登録する。
  - 案内文を UI に明記: 「対象リポジトリを 1 つに限定し、権限は Contents の Read/Write のみに
    してください」「トークンはこのブラウザの localStorage にのみ保存されます」
  - 保存先: `localStorage` キー `iwe-github-token`。リポジトリ指定(`owner/repo`)は作品ごとに
    `iwe.json` ではなく `localStorage` の `iwe-github-repo-<workId>` に保存する
    (トークンや連携設定を Git 管理対象のファイルに含めないため)。
- **アップロード(push 相当)**:
  1. `GET /repos/{owner}/{repo}/git/ref/heads/main` でリモート先端を取得
  2. リモート先端が前回同期時の SHA と一致していれば、ローカルのワークツリー全体から
     blob 群 → tree → commit を Git Data API(`POST .../git/blobs`, `/git/trees`, `/git/commits`)で作成し、
     `PATCH .../git/refs/heads/main` で先端を進める
  3. 成功したらリモート SHA を `localStorage`(`iwe-github-synced-<workId>`)に記録
  - ローカルの複数コミットを 1 コミットに畳んでよい(同期はバックアップ目的であり、
    細かい履歴はローカルに残っている)。コミットメッセージは「IWE sync: <日時>」+ ユーザー任意文。
- **ダウンロード(pull 相当)**:
  1. リモート先端の tree を再帰取得(`GET .../git/trees/{sha}?recursive=1`)し、各 blob を取得
  2. ローカルに未コミットの変更や未同期コミットがなければそのままワークツリーへ展開してコミット
  3. **競合時**(ローカルにもリモートにも前回同期以降の変更がある場合): ファイル単位で
     「ローカルを残す / リモートを取り込む」を選ばせる簡易マージ UI を出す。行単位の自動マージは
     v1 ではやらない。選択結果をコミットして同期 SHA を更新する
- **空リポジトリの初回連携**: リモートが空なら初回アップロードで `main` を作成する
  (`PUT /repos/{owner}/{repo}/contents/{path}` で 1 ファイル置いて ref を作る、等の手順を実装時に確認)。
- トークン未設定でも IWE は完全に使える(GitHub 連携はオプション機能)。

## 4. エディタと UI

### 4.1 画面構成(IDE 風 3 ペイン + ステータスバー)

```
+--------------------------------------------------------------+
| ヘッダー: ブランド / 作品切替 / スナップショット / GitHub同期 |
+----------+-----------------------------+---------------------+
| 左ペイン | 中央: エディタ              | 右ペイン            |
| 作品内   |  (textarea ベース)          |  プレビュー         |
| ツリー   |                             |  or 参照パネル      |
|          |                             |  or 履歴/差分       |
+----------+-----------------------------+---------------------+
| ステータスバー: 文字数 | 目標進捗 | 保存状態 | 変更あり表示  |
+--------------------------------------------------------------+
```

- **左ペイン**: メモ / プロット / 原稿 / 人物 の 4 セクションに分かれたファイルツリー。
  作成・リネーム・削除・(原稿の)並べ替え。現在ファイルをハイライト。
- **中央エディタ**: `textarea` ベース。既存ツール(echo/length/seam)と同じ
  「透明 textarea + backdrop オーバーレイ」パターンで、記法(ルビ・見出し・`[[リンク]]`)を
  軽くハイライトする。IME との相性を最優先し、リッチエディタ化はしない。
- **右ペイン**: タブで切替 —「プレビュー」(5 章)/「参照」(別ファイルを読み取り専用で並置。
  プロットを見ながら原稿を書く用途)/「履歴」(3.1)/「人物」(7.4)。閉じて中央を全幅にもできる。
- **モバイル**: 3 ペインは維持せず、左右ペインをドロワー化する。
  `shared/editor-mobile.css` の流儀に合わせる。

### 4.2 原稿記法(Nyuko 記法を踏襲)

パーサは `shared/nyuko-engine.js` のトークナイザを再利用する。対応記法:

| 記法 | 意味 | 対応状況 |
|---|---|---|
| `|漢字《かんじ》` / `漢字《かんじ》` | ルビ(青空文庫式) | nyuko-engine 実装済み |
| `[[rb:漢字 > かんじ]]` | ルビ(pixiv 式) | nyuko-engine 実装済み |
| 半角数字 2〜3 桁 | 縦中横(自動) | nyuko-engine 実装済み |
| `## 章題` / `### 節題` | 見出し | nyuko-engine 実装済み |
| `[break]` | 改ページ | nyuko-engine 実装済み |
| `《《強調》》` | **傍点(新規実装)** | pixiv 式。トークナイザに `kind: 'emphasis'` を追加 |

傍点は縦書きプレビューでは `text-emphasis: filled sesame`(縦書き時は文字の右)、
横書き系プレビューでは `text-emphasis: filled dot` で描画する。
nyuko-engine 本体への追加が大きくなる場合は、IWE 側のプリ/ポストプロセスで実装してもよい。

## 5. プレビュー(3 形式)

右ペインの「プレビュー」タブ内で 3 形式を切り替える。いずれも読み取り専用・現在ファイル
(原稿の場合は章単位)を対象とし、入力に追従して再描画(debounce 300ms 程度)する。

### 5.1 縦書き表示

- `shared/nyuko-engine.js` の DOM プレビュー(`paginate` + `createPageElement`、
  `writing-mode: vertical-rl`)をそのまま使う。判型は文庫相当をデフォルトに、
  `NyukoData.FONTS` のフォント切替も露出する。
- ルビ・縦中横・見出し・改ページが反映される。

### 5.2 原稿用紙風表示

- `genko/index.html` にある 20×20 マス目グリッド(`writing-mode: vertical-rl` +
  `grid-auto-flow: column` + `direction: rtl`)、禁則処理(句読点ぶら下げ・開き括弧追い出し)、
  縦組み字形変換(`VERT_FORMS` 相当)、三点リーダー回転のロジックを **`shared/genko-render.js`
  として切り出して共有**し、genko 本体と IWE の両方から使う形にリファクタする
  (genko 側の挙動を変えないこと。切り出しが困難なら v1 は必要部分のコピーでもよいが、その場合
  コピー元をコメントで明記する)。
- 1 枚 400 字で自動改ページし、「n 枚目 / 全 m 枚」を表示。

### 5.3 二段組表示

- 横書き二段組(文庫・同人誌の本文レイアウト風)。CSS `columns: 2` + `column-gap` ベースで、
  A5 想定の紙面枠にページ分割して表示する。段間罫線(`column-rule`)はオプション。
- ルビは `<ruby>` タグに変換して描画(nyuko トークナイザの出力を横書き HTML に変換する
  小さなレンダラを書く)。

## 6. 付加機能

### 6.1 文字数カウント・執筆目標/進捗

- ステータスバーに常時表示: 現在ファイルの文字数(空白・改行除く)/ 作品合計(原稿のみ合算)。
- 作品設定で目標文字数(`iwe.json` の `targetChars`)を設定でき、達成率バーを表示。
- **日別執筆量**: 1 日 1 回以上の粒度で「その日の作品合計文字数」を記録し、差分から日別の
  執筆量を出す。保存先は `localStorage` の `iwe-stats-<workId>`(JSON、`{ "2026-07-05": 12345, ... }`)。
  Git 管理には含めない(同期ノイズになるため)。統計パネルで直近 30 日の棒グラフ
  (依存ライブラリなし、div ベースで十分)を表示する。

### 6.2 全文検索

- ヘッダーの検索ボックス(ショートカット `Ctrl/Cmd+Shift+F`)。現在の作品内の全ファイルを
  インメモリで走査し、ヒット行をファイル別にグルーピングして表示。クリックで該当ファイルを
  開きその位置へスクロール。規模的に(数百ファイル・数 MB 程度)インデックスは不要。

### 6.3 相互リンク

- 本文中の `[[ファイル名]]` を作品内ファイルへのリンクとして解釈する
  (拡張子・ディレクトリ省略可。`title` またはファイル名のベース名で解決。曖昧なら候補を出す)。
- エディタの backdrop でリンクをハイライトし、`Ctrl/Cmd+クリック`(または長押し)で開く。
- 右ペイン「参照」タブの下部に **バックリンク一覧** を表示: 現在ファイルを `[[...]]` で参照している
  ファイル、および `source:` で現在ファイルを起点にしているファイル(2.4 の昇格リンク)。

### 6.4 登場人物・設定資料の管理

- 人物シート(2.3 の `type: character`)を右ペイン「人物」タブで一覧・閲覧できる。
  執筆中に名前・設定をすぐ確認する用途。
- `shared/cast-engine.js` の `TadekuCastEngine.analyzeCast(text)` を原稿全文に対して実行し:
  - **未登録の名前**: 抽出された人物名のうち、どの人物シートの `name` / `aliases` にも
    一致しないものを「シート未作成」として列挙 → ワンクリックでシート作成。
  - **表記ゆれ**: cast-engine の variants 警告をそのまま表示。
- 解析は重くなり得るため、人物タブを開いたときと明示的な「再解析」ボタンでのみ実行する。

## 7. 使わない・やらないこと(v1 スコープ外)

- テキスト・PDF 等へのエクスポート(将来拡張。Nyuko エンジンの `exportPdf` を流用すれば
  入稿 PDF まで出せる設計になっているため、後付けしやすい)
- Git のブランチ・行単位自動マージ・複数リモート
- 共同編集・リアルタイム同期
- 既存の `library/`(IndexedDB 原稿一覧)との相互運用(IWE は独立したデータ空間を持つ)

## 8. 実装ガイド(Composer 向け)

### 8.1 フェーズ分割

各フェーズ完了時点で動く状態を保つこと。

| フェーズ | 内容 | 完了条件 |
|---|---|---|
| 1 | LightningFS + 作品/ファイル CRUD + 3 ペイン UI + エディタ(自動保存・文字数) | 作品を作り、4 種のドキュメントを書いて再訪しても残っている |
| 2 | プレビュー 3 形式 + 記法ハイライト + 傍点記法 | ルビ・見出し入りの原稿が縦書き/原稿用紙/二段組で正しく表示される |
| 3 | ローカル Git(init/コミット/履歴/差分/巻き戻し) + スナップショット UI | スナップショットを 2 回取り、差分を見て、過去の状態に戻せる |
| 4 | GitHub 同期(PAT 設定・アップロード・ダウンロード・競合 UI) | 空の GitHub リポジトリと双方向に同期できる |
| 5 | 全文検索・進捗統計・相互リンク/バックリンク・人物管理(cast 連携) | 6 章の各機能が動作する |
| 6 | 昇格フロー(メモ→プロット→原稿)+ モバイル対応 + 仕上げ | 一連の創作フローがモバイルでも通しで使える |

### 8.2 新規作成するファイル(目安)

```

iwe/js/editor.js        … textarea + backdrop、自動保存、文字数

iwe/js/git.js           … isomorphic-git ラッパ(スナップショット・履歴・差分・復元)
iwe/js/github.js        … Git Data API 同期
iwe/js/search.js        … 全文検索
iwe/js/links.js         … [[リンク]] 解決・バックリンク
iwe/js/cast.js          … 人物管理 + cast-engine 連携
```

### 8.3 再利用する既存ファイル

| ファイル | 使い方 |
|---|---|
| `shared/nyuko-engine.js` / `shared/nyuko-data.js` | `<script>` で読み込み `window.NyukoEngine.paginate` / `createPageElement` / `formatManuscript` を縦書きプレビューと記法パースに使用 |

### 8.4 注意事項

- **IndexedDB / localStorage の命名**: IndexedDB は `iwe-fs` のみ新設。localStorage キーはすべて
  `iwe-` プレフィックス。既存の `tadeku-tools` DB・`shared/editor-prefs.js` には触れない。
- **既存ツールを壊さない**: `shared/` の既存ファイルへの変更は 5.2 の genko 切り出しのみ。
  それも genko の見た目・挙動を変えないこと(変更後に genko を目視確認する)。
- **XSS**: 本文をプレビューへ HTML 化する際は必ずエスケープしてからルビ等のタグを組み立てる
  (nyuko-engine の `esc` と同じ方針)。
- **PAT の扱い**: トークンを Git 管理対象ファイルや同期対象に絶対に含めない。console.log にも出さない。
- **パフォーマンス**: プレビュー再描画は debounce。cast 解析は明示操作時のみ。
  ファイル走査系(検索・バックリンク)は作品単位で、開いたときに実行。
- **文字数の定義**: 空白(半角/全角)・改行を除いた文字数で統一(既存エディタ群の慣習に合わせる)。
